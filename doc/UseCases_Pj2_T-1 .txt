Use Case 1: Upload Excel File
Author
Scheduling Committee Member


Primary Actor
Scheduling Committee Member


Priority
High


Status
Approved


Description
This use case allows the scheduling committee member to upload Excel files containing class scheduling data into the Bellini scheduling system.


Preconditions
•The system is running.
•The user has access to the upload interface.


Postconditions
•The uploaded Excel file is stored in the system.
•Class data is imported into the scheduling database.


Basic Flow
        1.The actor accesses the upload page.
        2.The actor selects an Excel file.
        3.The actor submits the file.
        4.The system uploads the file.
        5.The system imports the class data.
        6.The system confirms successful upload.


Alternative Flow
3a. If the file format is invalid
→ The system displays an error message and requests another file.
3b. If the selected file is blank/empty
→ The system displays an error message and does not import data.
3c. If the uploaded file is a duplicate of an existing uploaded file
→ The system skips saving a duplicate copy and confirms duplicate detection while still processing import rules.
3d. If imported class entries match existing entries by CRN and semester
→ The system keeps the latest uploaded version for that class instance instead of creating duplicate schedule rows.
6a. The actor may remove an uploaded Excel file from the uploaded files list.
















Use Case 2: Search and Filter Classes
Author: MayfairMI6
Primary Actor : Scheduling Committee Member
Priority: High ( quickly locating specific classes is a core day-to-day need for the committee)
Status Approved
Description This use case allows a scheduling committee member to search for specific classes in the Bellini scheduling system using one or more filter criteria, such as course code, instructor name, semester, or department.
Preconditions
* The system is running.
* At least one semester's schedule data has been imported into the system.
* The user has access to the search page.
Postconditions
* The system displays a list of classes matching the entered criteria.
* If no matches are found, the system notifies the user that no results were returned.
Basic Flow
1. The actor navigates to the Search & Filter Classes page.
2. The actor enters one or more search criteria — such as a course code, instructor name, semester, or department.
3. The actor clicks "Search."
4. The system retrieves all classes that match the provided criteria.
5. The system displays the results, showing details such as course title, section, instructor, room, meeting days and times, semester, and enrollment.
Alternative Flow
3a. If no criteria are entered before clicking "Search": → The system does not perform a search and prompts the user to enter at least one search value.
5a. If no classes match the entered criteria: → The system displays a message indicating that no results were found, and the actor may adjust their search and try again.

Use Case 3: Compare Semester Schedules
Author : MayfairMI6
Primary Actor :SchedulingCommittee Member
Priority Medium ( valuable for planning but not required for everyday scheduling tasks)
Status Approved
Description This use case allows a scheduling committee member to compare course offerings and enrollment data between two different semesters, helping identify which courses were added, removed, or carried over.
Preconditions
* The system is running.
* Schedule data for at least two different semesters has been imported into the system.
* The user has access to the comparison page.
Postconditions
* The system displays a side-by-side summary of the two selected semesters, including statistics and categorized course lists.
* If the actor selects a specific course, the system displays detailed section-level information for that course in both semesters.
Basic Flow
1. The actor navigates to the Compare Schedules page.
2. The actor selects a first semester and a second semester from the available options.
3. The actor clicks "Compare Schedules."
4. The system generates a comparison showing:
   * Courses offered only in the first semester.
   * Courses offered only in the second semester.
   * Courses offered in both semesters.
   * Summary statistics for each semester, including total classes, unique courses, and total enrollment.
5. The system displays the comparison results.


Alternative Flow
2a. If the actor selects the same semester for both fields: → The system alerts the user that two different semesters must be chosen, and the comparison is not submitted.
2b. If the actor does not select a semester for one or both fields: → The system alerts the user to complete both selections before proceeding.
4a. If one of the selected semesters has no data in the system: → The system treats it as having no courses. All courses from the other semester appear as exclusive to that semester, and the empty semester's statistics display as zero.
5a.The actor clicks on a specific course to view a detailed breakdown, showing the number of sections, total enrollment, and individual section details for each semester, along with a short summary of what changed.








Use Case 4: ViewInstructorSchedule
Use Case Thumbnail: UC03-ViewInstructorSchedule
Use Case Description: This use case describes the viewing of a specific instructor's weekly teaching schedule for a selected semester, including automatic detection of time conflicts between their assigned classes.
Preconditions: At least one semester's class data has been imported into the scheduling database. The system is running and accessible.
Postconditions: The instructor's weekly schedule is displayed as a time-slot grid. Any scheduling conflicts between the instructor's classes are identified and highlighted.
Actors: A10-SchedulingCommitteeMember
Use Case Relationships: Associated with actor A10-SchedulingCommitteeMember (via browser). Depends on UC01-UploadExcelFile for schedule data to be available.


Basic Flow:
1. The actor navigates to the home page and selects "Instructor Weekly Schedule."
2. The system retrieves the list of available semesters from the database and displays them in a dropdown.
3. The actor selects a semester from the dropdown.
4. The system retrieves the list of instructors who have classes in the selected semester and populates the instructor dropdown.
5. The actor selects an instructor from the dropdown.
6. The actor submits the selection by pressing "View Schedule."
7. The system retrieves all classes assigned to the selected instructor for the chosen semester.
8. The system parses the meeting days and meeting times for each class.
9. The system builds a weekly timetable grid organized by time slot (rows) and day of the week (columns).
10. The system checks for overlapping class times on the same day and flags any conflicts .
11. The system displays the weekly schedule grid, total class count, and conflict status to the actor.
Alternative Flow:
If overlapping class times are detected in step 10, the system displays a conflict warning section above the timetable listing each conflict with the day name, course codes, and overlapping time ranges. Conflicting class blocks in the grid are highlighted in red.
Exceptions:
- If no instructors exist for the selected semester (step 4), the system displays an empty instructor dropdown and the actor must select a different semester.
- If the instructor has no classes with valid meeting days or times, the system displays a message: "No classes scheduled for this instructor in the specified semester"
Constraints:
- Cross-listed UG/GR pairs (same course name, with 4000-4999 as undergraduate and 5000+ as graduate) must not be flagged as scheduling conflicts when their times overlap.
- Entries representing the same class instance (same CRN) must not be treated as scheduling conflicts.
User Interface Specifications: UI03-InstructorScheduleForm consisting of home.html, instructor_select.html, and instructor_schedule.html. The home page provides navigation, the selection page presents semester and instructor dropdowns, and the schedule page renders the weekly timetable grid with conflict indicators.
Metrics: Simple
Priority: High
Status: Implemented
Author & History: Anthony Saade




Use Case 5:


        Use Case Thumbnail: UC50-Vacancy Search        


Actors: A10-SchedulingCoordinator        


        Use Case Description:
This use case deals with the actions a scheduling committee member will have to take when searching for open time slots for a new class in a specified semester. The member will have to specify the length of the class and which semester it will be applied in, though they may also filter the results to a specific room.        


        
        Use Case Relationships:


Preconditions:
        The website has data of each semester and their course schedules.
        The backend can search and filter through the provided course schedules


        Basic Flow:
1. Enter Site
2. Navigate to Vacancy Search page
3. Obtain Valid Rooms and Semesters from database
4. Create Search Form from obtained data
5. Actor inputs their search specifications
6. Navigate to the Vacancy Results page
7. Using the actor’s input, find the specified semester and room
8. Obtain current time table from valid rooms of specified semester
9. Using actor’s input search for open time slots that are as long as they desired
10. Send the list of valid rooms and their timeslots to the Vacancy Results page
11. Display the list as a table to the actor
        
Alternative Flow
11a. If the list provided by the backend is empty, state that there are no vacant spots in the Vacancy Results page


        User Interface Specifications: home.html, VacancySelect.html, VacancyTable.html
Three working webpages that will hold the option to use the function, parameters before using the function and the results of the function respectively.
        
Priority: Medium.
        This function is mainly an easier way to see valid time slots for a scheduling member to use without needing to check through the regular schedule search. That said it is still a main function that can be used from the home site, making a somewhat fundamental action.
        
Status: Implemented
        
Author and History: Jose Wong
