

UML DIAGRAM REQUIREMENTS FOR BELLINI CLASSES PROJECT

GENERAL RULES (APPLIES TO ALL DIAGRAMS)
1. Use standard UML notation.
2. Keep diagrams high-level (no internal system implementation details unless explicitly required).
3. Maintain consistency in naming (actors, use cases, actions).
4. Reflect only documented requirements from use case descriptions.
5. Prefer clarity over complexity.
6. Make minimal changes when updating diagrams.

--------------------------------------------------

USE CASE DIAGRAM REQUIREMENTS

1. Diagram Type:
   - UML Use Case Diagram

2. Actors:
   - Scheduling Committee Member
   - Scheduling Coordinator

3. System Boundary:
   - Rectangle labeled "Bellini Classes System"

4. Documented Use Cases (must be included):
   - Upload Excel File
   - Remove Uploaded File
   - Search and Filter Classes
   - Compare Semester Schedules
   - View Instructor Schedule
   - Vacancy Search

5. Inferred Use Cases (optional but user-goal level only):
   - View Class Details
   - Validate Uploaded File
   - Detect Duplicate Upload File
   - Detect Scheduling Conflicts
   - View Room Schedule
   - View Semester Statistics

6. Color Rules:
   - Documented use cases → LightGreen
   - Inferred use cases → LightYellow

7. Relationships:
   - Actor to use case → association (--)
   - Use case to use case → <<include>> only if clearly justified
   - Avoid non-standard relationships (e.g., "depends on")

8. Layout:
   - Actors outside system boundary
   - Use cases inside system boundary
   - Keep layout simple and readable (similar to lecture slides)

9. Additional Requirement:
   - Include a legend explaining color usage
   - Include user story → use case mapping at the bottom

--------------------------------------------------

ACTIVITY DIAGRAM REQUIREMENTS

1. Diagram Type:
   - UML Activity Diagram (based on Chapter 7 examples)

2. Structure:
   - Include start node and end node
   - Use actions (rounded rectangles)
   - Use decision nodes (diamonds) for branching
   - Use directed edges for flow

3. Swimlanes:
   - Separate lanes for:
     • Actor (e.g., Scheduling Committee Member)
     • System

4. Content Source:
   - Must be derived directly from:
     • Basic Flow
     • Alternative Flow(s)
     • Exception(s)
   - Do NOT invent steps not in use case

5. Behavior Rules:
   - Model interaction between actor and system only
   - No internal system objects or technical details

6. Decision Handling:
   - Each alternative flow must appear as a decision branch
   - Clearly label Yes/No branches

7. Flow Quality:
   - Must be sequential and logically correct
   - Avoid redundant or duplicated actions

8. Termination:
   - All paths must end in a stop node

9. Style:
   - Keep diagrams clean and readable
   - Follow structure similar to Figures 7.2–7.5 from Chapter 7

10. Domain-Specific Constraints (Bellini Classes):
   - Courses can be cross-listed between undergraduate and graduate levels.
   - Course numbering rule:
       • 4000–4999 → Undergraduate courses
       • 5000 and above → Graduate courses
   - If two courses have the same name but different levels, they should be treated as cross-listed.
   - Cross-listed pairs (same course name across UG/GR levels) should not be flagged as instructor schedule conflicts when they overlap in time.
    - Duplicate rows representing the same class instance (same CRN) must not be treated as scheduling conflicts.
    - Upload workflows must include decision branches for:
       • Missing file field
       • Empty file content
       • Invalid/non-Excel file format
       • Duplicate uploaded file detection (no duplicate copy saved)
    - File management workflows should include a remove uploaded file action from the uploaded files list.
   - Activity diagrams involving schedules (e.g., instructor schedule, comparison, vacancy search) must:
     • Account for cross-listed courses as a single logical course
     • Ensure no instructor time conflicts occur due to cross-listed entries
     • Clearly represent detection or handling of such conflicts in the flow when applicable
